RedString Dialogue API AWS Deployment Steps
==========================================

Recommended setup
-----------------
1. Use one EC2 g4dn.xlarge Spot instance.
2. Use an AWS Deep Learning GPU AMI.
3. Use the GGUF model:
   bartowski/Llama-3.2-3B-Instruct-GGUF
   file: Llama-3.2-3B-Instruct-Q4_K_M.gguf
4. Keep the model on EBS.
5. Keep the service warm with REDSTRING_WARM_START=true.

Why this is the best cold-start strategy
----------------------------------------
1. Do not scale to zero for demo usage.
2. Do not redownload the model on every restart.
3. Keep one process always running on one instance.
4. If you want even less startup time, bake the GGUF into a custom AMI.

Step 1: Provision infrastructure
--------------------------------
From backend/iac/terraform:

terraform init
terraform apply \
  -var="ami_id=<gpu_dlami_id>" \
  -var="ssh_key_name=<your_keypair>" \
  -var="operator_cidr=<your_public_ip>/32"

This creates:
- EC2 instance
- security group
- IAM role
- spot settings

Step 2: Copy the backend to the instance
----------------------------------------
scp -i <key.pem> -r backend ubuntu@<PUBLIC_IP>:/opt/redstring/

Step 3: Configure environment
-----------------------------
ssh -i <key.pem> ubuntu@<PUBLIC_IP>
sudo cp /opt/redstring/backend/deploy/redstring-dialogue.env.example /etc/redstring-dialogue.env
sudo nano /etc/redstring-dialogue.env

Set at minimum:
- REDSTRING_SECRET_KEY
- REDSTRING_LLM_CONFIG=/opt/redstring/backend/config/local_llm_config.json
- REDSTRING_HF_REPO_ID=bartowski/Llama-3.2-3B-Instruct-GGUF
- REDSTRING_HF_FILENAME=Llama-3.2-3B-Instruct-Q4_K_M.gguf
- REDSTRING_WARM_START=true

Optional alternatives:
- REDSTRING_MODEL_S3_URI=s3://bucket/path/model.gguf
- REDSTRING_MODEL_URL=https://host/path/model.gguf

Step 4: Install the service
---------------------------
chmod +x /opt/redstring/backend/start_backend.sh
chmod +x /opt/redstring/backend/deploy/install_on_ec2.sh
sudo APP_ROOT=/opt/redstring /opt/redstring/backend/deploy/install_on_ec2.sh

Step 5: Verify
--------------
curl http://<PUBLIC_IP>:8000/health

Warm the model explicitly:

curl \
  -X POST \
  -H "Authorization: Bearer <REDSTRING_SECRET_KEY>" \
  http://<PUBLIC_IP>:8000/warmup

Example dialogue request:

curl \
  -H "Authorization: Bearer <REDSTRING_SECRET_KEY>" \
  -H "Content-Type: application/json" \
  -d @payload.json \
  http://<PUBLIC_IP>:8000/dialogue

Step 6: Service operations
--------------------------
sudo systemctl status redstring-dialogue
sudo journalctl -u redstring-dialogue -f
sudo systemctl restart redstring-dialogue

How the API response shape is guaranteed
----------------------------------------
1. The prompt asks for JSON only.
2. The backend parses the model output.
3. It accepts output only if the JSON object has exactly:
   - response
   - clues_unlocked
4. If parsing fails, it falls back to deterministic grounded generation.
5. FastAPI then serializes the final response using the typed API response model.

Notes
-----
1. The API now has no ElevenLabs/TTS integration.
2. The API no longer uses an app-layer IP allowlist.
3. Security is bearer-token based.
4. The game should call POST /warmup on startup if /health reports llm_ready=false.
