========================================
RedString Dialogue API System Design
========================================

Purpose
-------
This backend is a standalone AI dialogue API for RedString.
It is not the full game backend.

Input
-----
The game client sends:

1. npc_id or character_info
2. player_question
3. evidence_id
4. generation_backend
5. game_state
   - found_clues
   - asked_questions
   - npc_id

Output
------
The API returns:

{
  "response": "NPC dialogue response",
  "clues_unlocked": ["EVID_XX"]
}

Core Flow
---------
1. Authenticate request with Bearer token.
2. Check confession trigger conditions.
3. Attempt semantic retrieval over pregenerated dialogue.
4. If retrieval misses, choose a generation backend:
   - local llama.cpp generation
   - Gemini API generation
   - deterministic fallback if model output is invalid or no model backend is available
5. Validate and sanitize response.
6. Filter clues so only legal evidence IDs survive.
7. Return typed API response.

Grounding Rules
---------------
The generator may only use:
- personality_prompt
- behavior_guidelines
- knowledge_base
- relationship_to_victim
- evidence_knowledge

The system does not trust raw model output directly.
If the model does not return valid JSON with exactly:
- response
- clues_unlocked

then the backend falls back to deterministic grounded generation.

Clue Rules
----------
1. Only clues already present in character_info.evidence_knowledge may be returned.
2. No duplicate clues.
3. No already-found clues returned again.
4. No new evidence IDs may be invented.

Confession Rule
---------------
If all evidence IDs referenced by confession_trigger are present in
game_state.found_clues, the confession response overrides normal routing.

Deployment Shape
----------------
Recommended AWS deployment:
- one EC2 g4dn.xlarge Spot instance
- one quantized GGUF model
- one always-on process
- model stored on EBS
- REDSTRING_WARM_START=true

Recommended Model
-----------------
Repo:
bartowski/Llama-3.2-3B-Instruct-GGUF

File:
Llama-3.2-3B-Instruct-Q4_K_M.gguf

Cold Start Strategy
-------------------
Best practical approach:
1. Keep one instance running during the demo.
2. Store the GGUF on EBS instead of redownloading on boot.
3. Warm the model when the service starts.
4. If needed, bake the model into a custom AMI.

Security
--------
Requests are protected with:
- Authorization: Bearer <REDSTRING_SECRET_KEY>

The API is network-open by default at the infrastructure layer.
Security is handled by the bearer token.

Remote Generation Option
------------------------
If REDSTRING_GEMINI_API_KEY is configured, the backend can use Gemini as the
generation backend when:
- the request sets generation_backend=gemini
- or generation_backend=auto and the local model has not spun up yet
